# AI Company 강의 — 데모 녹화 스토리보드 v3

이 문서는 `AI-Company-lecture-v4-polished-slides.md`에 맞춘 녹화 대본이다.

## 원칙

- 앱 소개가 아니라 강의 논지를 보여준다.
- 한 데모는 45-90초로 제한한다.
- 정지 화면 설명을 쓰지 않는다. 실제 앱 실행 흐름을 보여준다.
- 화면과 음성은 같은 대상을 가리켜야 한다.
- 정보가 많은 화면은 여러 비트로 나눈다.
- 자막은 전체 문장 받아쓰기가 아니라 의미 캡션으로 쓴다.
- 기존 구버전 녹화는 최종본에 쓰지 않는다.

## 데모 순서

| 순서 | 앱 | 역할 | 권장 길이 |
|---|---|---|---|
| 1 | ResearchPilot Academy | 에이전트가 연구 과정을 수행한다 | 70-90초 |
| 2 | Proposal Studio / Plan Builder | 선택 부담과 보이지 않는 자동화의 불안 | 45-60초 |
| 3 | InsightValidationServer | 그럴듯함과 공개 가능성의 차이 | 60-75초 |
| 4 | KWCS-QC 또는 ResearchPilot Data | 실제 문제가 사용 이유를 만든다 | 60-75초 |
| 5 | Press Release Builder / PR Studio | 번역 문제가 아니라 프레임 전환 | 60-80초 |
| 6 | PR Studio feedback | 고차원 선택을 포함한 자동화 | 60-80초 |

# Demo 1 · ResearchPilot Academy — 연구 과정 전체 수행

- **삽입 위치**: 슬라이드 `연구 과정 전체를 수행한다` 직후
- **앱 폴더**: `C:\git-app\TextAnalysisCopilot`
- **권장 URL**: `http://127.0.0.1:5186/?demo=conference&view=construct`
- **목표 길이**: 70-90초
- **한 줄 메시지**: 에이전트는 대화창 답변이 아니라 연구 과정을 이어 간다.

> 녹화 전 확인: 현재 화면 상단 이름이 `Text Analysis Copilot`로 보일 수 있다. 강의에서는 ResearchPilot Academy로 소개하므로, 화면 제목을 바꾸거나 상단을 덜 강조하는 구도로 촬영한다.

## 구조

1. 연구문제와 데이터가 들어온다.
2. 앱이 연구 개념, 분석 단위, 코드북, 메시지 추출 계획을 만든다.
3. 분석 결과와 근거 문장이 나온다.
4. 검증과 보고서로 이어진다.
5. 의미: 대화창 답변이 아니라 연구 절차의 자동 수행이다.

## 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|---|---|---|
| 0:00 | construct 화면 시작 | 연구질문과 분석 단위 | "연구문제와 데이터가 들어오면, 앱은 먼저 분석할 일을 정리합니다." |
| 0:12 | intake 또는 construct 항목 확인 | central concept, RQ, unit | "무엇을 분석할지, 어떤 단위로 볼지 정합니다." |
| 0:24 | messages 또는 design 흐름으로 이동 | 메시지/코드북 생성 영역 | "대화창 답변이 아니라, 분석 절차가 만들어집니다." |
| 0:40 | content 또는 frame 결과 화면 | 코드 분포와 근거 문장 | "분석 결과는 숫자만이 아니라 근거 문장과 함께 나옵니다." |
| 0:56 | validation 또는 report 화면 | 검증/리포트 버튼과 결과 | "마지막에는 검증과 보고서로 이어집니다." |
| 1:10 | 흐름 전체가 보이는 화면 hold | 현재 프로젝트 상태 | "핵심은 하나의 답이 아니라, 연구 과정 전체를 수행한다는 점입니다." |

## 자막

- 연구문제와 데이터를 받는다.
- 분석 단위와 코드북을 정한다.
- 결과와 근거를 함께 낸다.
- 검증과 보고서로 이어진다.

# Demo 2 · Proposal Studio / Plan Builder — 선택 부담과 자동화 불안

- **삽입 위치**: 슬라이드 `단계가 많으면 부담이 된다`와 `그러나 보이지 않으면 불안하다` 사이
- **목표 길이**: 45-60초
- **한 줄 메시지**: 선택지는 부담이고, 보이지 않는 자동화는 불안이다.

## 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|---|---|---|
| 0:00 | Proposal Studio 단계 화면 | 8단계 흐름 | "처음에는 체계적인 8단계 흐름을 만들었습니다." |
| 0:12 | 많은 선택 항목 강조 | 옵션 영역 | "하지만 막상 쓸 때는 선택지가 부담이 됩니다." |
| 0:24 | Plan Builder 전자동 흐름 | 자동 실행 버튼 | "그래서 자동화했습니다." |
| 0:38 | 판단 근거가 보이는/부족한 화면 | 기준·근거 영역 | "그런데 자동화는 AI가 사람 대신 선택하는 일입니다. 근거가 안 보이면 불안합니다." |

## 자막

- 선택지가 많으면 일이 줄지 않는다.
- 자동화는 편하지만 불안할 수 있다.
- 판단 근거가 보여야 한다.

# Demo 3 · InsightValidationServer — 그럴듯함과 공개 가능성

- **삽입 위치**: 슬라이드 `그럴듯함과 공개 가능성은 다르다` 직후
- **앱 폴더**: `C:\git-app\InsightValidationServer`
- **권장 URL**: `http://127.0.0.1:4020/`
- **목표 길이**: 60-75초
- **한 줄 메시지**: 생성과 검증은 분리되어야 한다.

## 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|---|---|---|
| 0:00 | 그럴듯한 분석 요약 또는 검증 입력 | 요약/입력 패널 | "결과는 그럴듯할 수 있습니다." |
| 0:12 | `cell_sum_001` 로드 | 케이스 선택 | "하지만 그럴듯함과 공개 가능성은 다릅니다." |
| 0:24 | 검증 실행 | 실행 버튼과 진행 | "따로 검증합니다." |
| 0:38 | `BASE_N_MISMATCH` | N 불일치 근거 | "기준 N과 결과 N이 맞지 않습니다." |
| 0:50 | `CELL_SUM_MISMATCH` | 셀 합계 근거 | "셀 합계도 표 전체와 맞지 않습니다." |
| 1:02 | `do_not_release` | 판정 배지 | "그래서 공개 전 수정이 필요합니다." |

## 자막

- 그럴듯함과 공개 가능성은 다르다.
- 기준 N과 셀 합계를 대조한다.
- 정합성이 맞지 않으면 공개하지 않는다.
