# AI Company 강의 — 데모 녹화 스토리보드 v2

이 문서는 강의 슬라이드(`lectures/courses/AI-Company-lecture/AI-Company-lecture-v2-slides.md`) 사이에 삽입할 데모 영상의 **촬영 대본**이다.

## 원칙 (production-memory 준수)

- 기존 녹화본(demo-01~05 구버전)은 최종에 쓰지 않는다. 새로 녹화한다.
- 각 데모는 30–90초 독립 클립. 한 데모에 메시지 하나.
- 정적 화면 나열 금지. 앱이 **실제로 실행되는** 장면을 담는다: 입력 → 실행 → 결과 → 멈춰서 볼 판단 지점 → 의미.
- 슬라이드에는 '볼 것/DEMO' 같은 메타 문구를 쓰지 않는다. 데모는 해당 논점 직후 자연스럽게 재생한다.
- 화면 비율 16:9, 녹화 해상도 1920×1080(슬라이드와 동일 프레임). 커서 하이라이트 ON, 판단 지점에서 줌 크롭.
- 경로: 소스 앱은 로컬, 출력 mp4는 Dropbox(`gitwork_data/AI Company_lecture/demos/`). mp4는 git에 커밋하지 않는다.

# Demo 1 · Text Analysis Copilot — 수정 수준 판단

- **출력 파일**: `demos/demo-01-text-analysis-repair.mp4`
- **삽입 위치**: 슬라이드 "수정 수준이 중요하다" 직후
- **목표 길이**: 75–90초
- **한 줄 메시지**: AI는 "수정한다"가 아니라 **"어느 수준을 수정할지"** 판단한다.

### 구조 (문제 → 실행 → 판단지점 → 의미)

1. **문제상황** — 분석 결과가 약하다. 보통은 그냥 다시 돌린다.
2. **실행** — Autonomous repair를 실행한다.
3. **판단지점(핵심)** — 앱이 수정 층위를 스스로 올린다: 코딩 규칙 → 코드북 → 이론/모형 → 연구 질문. 어디서 멈추는지, 왜 그 수준인지 근거를 보여준다.
4. **의미** — 단순 재생성이 아니라 수정 수준 선택. 그 근거가 보인다.

### 비트 (timecode / 화면 동작 / 캡처 포커스 / 내레이션)

| t | 화면 동작 | 캡처 포커스 | 내레이션(짧게) |
|---|-----------|-------------|----------------|
| 0:00 | reflection 뷰, 약한 분석 결과 카드 표시 | 결과 카드 전체 | "분석 결과가 약합니다. 보통은 그냥 다시 돌립니다." |
| 0:08 | `Autonomous repair` 클릭 | 버튼 클릭 → 진행 표시 | "이 앱은 다시 돌리기 전에, 어디를 고칠지 먼저 봅니다." |
| 0:18 | 1단계: 코딩 규칙 수정 로그 | 규칙 diff 영역 줌 | "먼저 코딩 규칙입니다. 표현 수준의 수정." |
| 0:32 | 2단계: 코드북/분류 재검토 | 코드북 패널 줌 | "그래도 약하면 코드북으로 올라갑니다." |
| 0:46 | 3단계: 이론/모형 또는 연구질문 재검토 표시 | 수준 라벨 강조 | "여기서 멈추지 않고, 이론과 질문 수준까지 판단합니다." |
| 1:00 | 최종: 선택된 수정 수준 + 근거 요약 | 근거 요약 카드 정지 | "핵심은 수정이 아니라, 어느 수준을 고칠지 정한 것입니다." |
| 1:12 | 정지 프레임 hold 3초 | 수준 사다리 | (무음) |

### 멈춰서 볼 지점
수정 수준이 한 단계 올라가는 전환 프레임 1곳을 2초 hold + 줌. "코딩 규칙 → 코드북" 전환이 가장 분명하면 그 지점.

# Demo 2 · PR Studio — 보도자료 판단 구조

- **출력 파일**: `demos/demo-02-pr-studio-press-release.mp4`
- **보조 이미지**: `screens/assets/pr-studio-press.png`
- **삽입 위치**: 슬라이드 "문장은 좋아졌지만 보도자료는 아니었다" 직후
- **목표 길이**: 60–80초
- **한 줄 메시지**: 보도자료는 좋은 문장이 아니라 **배포 가능한 메시지 구조**다. (번역 → 재작성 → 메시지 재구성)

### 구조

1. **문제상황** — 번역·환각·어색한 영어를 다 고쳤는데도 보도자료로는 부족하다.
2. **실행** — PR Studio에서 같은 원문을 보도자료 구조로 재구성한다.
3. **판단지점** — 대상(누구에게), 목적(무엇을 바꾸나), 톤, 배포 가능성, 언론 대응 포인트가 함께 잡히는 화면.
4. **의미** — 문장 라이브러리로 안 되는 업무 판단 구조. 문제의 틀을 바꿨다.

### 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|-----------|-------------|----------|
| 0:00 | 직역/매끄러운 영문 초안 표시 | 초안 텍스트 | "문장은 매끄럽습니다. 그런데 보도자료로는 약합니다." |
| 0:12 | 보도자료 재구성 실행 | 실행 버튼 | "번역을 더 고치는 대신, 보도자료로 다시 씁니다." |
| 0:24 | 대상·목적·톤 패널 채워짐 | 판단 패널 줌 | "대상이 누구인지, 무엇을 바꾸려는지 먼저 잡습니다." |
| 0:40 | 메시지 구조(리드·핵심·인용·배포) 생성 | 구조 블록 | "보도자료는 문장이 아니라 메시지 구조입니다." |
| 0:54 | 배포 가능성/언론대응 체크 | 체크 항목 | "배포해도 되는지, 책임질 수 있는 표현인지 봅니다." |
| 1:06 | 완성 보도자료 정지 hold | 전체 | "번역을 고친 게 아니라, 문제의 틀을 바꿨습니다." |

### 멈춰서 볼 지점
"대상·목적" 패널이 채워지는 순간 — 문장이 아니라 판단이 들어가는 지점.

# Demo 3 · PR Studio — 유튜브 피드백 기반 수정 수준

- **출력 파일**: `demos/demo-03-pr-studio-feedback-repair.mp4`
- **삽입 위치**: 슬라이드 "자동화는 판단의 수준을 포함한다" 직후
- **목표 길이**: 70–90초
- **한 줄 메시지**: 피드백을 보고 **무엇을 고칠지(제목/썸네일/설명/프레임/타깃)** 판단한다. 가드레일은 과잉 수정을 막는다.
- **설정**: `?tab=performance`

### 구조

1. **문제상황** — 영상 성과가 기대보다 낮다. 무작정 다시 만들면 안 된다.
2. **실행** — 성과 수집 + 댓글/지표 피드백 입력.
3. **판단지점** — 앱이 수정 수준을 제안: 제목만? 썸네일 약속? 설명 구조? 메시지 프레임? 타깃? + 가드레일 경고.
4. **의미** — 자동 실행이 아니라, 어느 차원의 문제인지 판단하는 자동화.

### 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|-----------|-------------|----------|
| 0:00 | performance 탭, 낮은 지표 | 지표 카드 | "성과가 낮습니다. 그냥 다시 만들기 쉽습니다." |
| 0:10 | `성과 수집` 클릭 | 수집 진행 | "먼저 댓글과 지표를 모읍니다." |
| 0:22 | 피드백 입력/반영 | 입력 영역 | "반응을 근거로 넣습니다." |
| 0:34 | 수정 수준 후보 제시 | 후보 리스트 줌 | "제목인가, 썸네일 약속인가, 설명 구조인가, 메시지 프레임인가, 타깃인가." |
| 0:50 | 가드레일 경고 표시 | 경고 배지 | "가드레일은 댓글 몇 개로 전략을 뒤집지 않게 막습니다." |
| 1:04 | 선택된 수정 수준 + 다음 실험 | 결정 카드 hold | "자동으로 하되, 어느 차원을 고치는지 판단합니다." |

### 멈춰서 볼 지점
수정 수준 후보가 펼쳐지는 화면 — "다음 실험"이 정해지는 순간.

# Demo 4 · Insight Validation Server — 검증 화면

- **출력 파일**: `demos/demo-04-validation-server-ui.mp4`
- **삽입 위치**: 슬라이드 "그럴듯한 결과를 검증한다" 직후
- **목표 길이**: 45–60초
- **한 줄 메시지**: **생성과 검증은 분리**된다. 그럴듯함이 아니라 공개 가능 상태인지 본다.
- **설정**: 데모 케이스 `cell_sum_001`

### 구조

1. **문제상황** — 분석 요약이 그럴듯하다.
2. **실행** — 같은 결과를 검증 서버에 올린다.
3. **판단지점** — 검증이 코드·데이터·결과·해석을 따로 대조한다(점수 계산이 아님).
4. **의미** — 생성기와 검증기는 별개 절차.

### 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|-----------|-------------|----------|
| 0:00 | 그럴듯한 분석 요약 표시 | 요약 카드 | "결과는 그럴듯합니다." |
| 0:10 | `cell_sum_001` 케이스 로드 → 검증 실행 | 실행 버튼 | "그럴듯함과 공개 가능성은 다릅니다. 따로 검증합니다." |
| 0:22 | 검증 항목 진행(기준 N·셀 합계·분석 단위·공개 기준) | 항목 리스트 | "코드, 데이터, 결과, 해석을 대조합니다." |
| 0:38 | 검증 진행 완료 직전 정지 | 항목 패널 hold | "점수 계산이 아니라, 업무 기준과 맞는지 봅니다." |

### 멈춰서 볼 지점
검증 항목 리스트가 한 번에 보이는 프레임. (결과 판정은 Demo 5로 이어짐)

# Demo 5 · Insight Validation Server — 검증 결과

- **출력 파일**: `demos/demo-05-validation-result.mp4`
- **삽입 위치**: 슬라이드 "정합성이 공개 가능성을 만든다" 직후 (Demo 4에 바로 이어짐)
- **목표 길이**: 45–60초
- **한 줄 메시지**: **정합성**이 공개 가능성을 만든다 — 막는 근거를 명시한다.

### 구조

1. **문제상황** — 검증을 돌렸다.
2. **실행** — 결과 판정 화면.
3. **판단지점(핵심)** — `BASE_N_MISMATCH`, `CELL_SUM_MISMATCH` → `do_not_release`. 왜 막혔는지 근거가 명시된다.
4. **의미** — 정합성은 만능 정답은 아니지만, 학술·조사 업무의 최소 사용가능성 장치.

### 비트

| t | 화면 동작 | 캡처 포커스 | 내레이션 |
|---|-----------|-------------|----------|
| 0:00 | 검증 결과 판정 표시 | 판정 헤더 | "결과가 나왔습니다." |
| 0:10 | `BASE_N_MISMATCH` 항목 펼침 | 코드+설명 줌 | "기준 N과 결과 N이 다릅니다." |
| 0:22 | `CELL_SUM_MISMATCH` 항목 펼침 | 셀 합계 비교 줌 | "셀 합계가 표 전체와 맞지 않습니다." |
| 0:34 | `do_not_release` 배지 강조 | 배지 hold 3초 | "그래서 공개 전 수정이 필요합니다." |
| 0:46 | 정지 프레임 | 근거 3종 한 화면 | "정합성이 맞아야 공개 가능한 결과입니다." |

### 멈춰서 볼 지점
`do_not_release` 배지 + 두 개의 mismatch 근거가 한 화면에 보이는 프레임.
