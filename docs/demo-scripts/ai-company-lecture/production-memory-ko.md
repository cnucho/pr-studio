# AI Company Lecture Production Memory

## Decision

PR Studio should manage this lecture as an external demo/lecture production project, but the full lecture folder should not be moved into the PR Studio repo.

Keep these boundaries:

- Lecture source repo: `C:\git-app\lectures\courses\AI-Company-lecture`
- Large generated assets: `C:\Users\ciadmin\Dropbox\gitwork_data\AI Company_lecture`
- PR Studio project notes and production rules: `docs/demo-scripts/ai-company-lecture`

The reason is simple: PR Studio is the production and improvement workspace, while the lecture folder contains source material and large generated media. Moving the whole folder into PR Studio would mix ownership, bloat Git, and make it harder to reuse the lecture material elsewhere.

## Current Problem

The current lecture video narration is not natural enough because it is generated by mechanically combining slide titles, bullet text, and notes.

Observed issues:

- Titles become awkward spoken sentences, such as `이 강의는 기술 자랑이 아니다입니다`.
- Bullet lists are read as flat text without transitions.
- Demo slides say that a separate demo should be checked, instead of guiding the viewer through what to watch.
- Some technical paths and asset labels can leak into narration.
- The voice-screen relationship is weak because the narration was not written as timed beats.
- Existing demo videos should not be reused as final lecture demos. They are too static and the narration does not sound natural enough.

This is not only a TTS provider problem. Google TTS can sound much better when the input is a spoken script. The main limitation is the current script generation method.

## Better Production Model

Use three layers instead of one slide-to-voice conversion:

1. Slide layer
   - What appears on screen.
   - Keep concise and visually scannable.

2. Spoken-script layer
   - What the lecturer or TTS voice actually says.
   - Use natural Korean speech, short sentences, transitions, examples, and pauses.

3. Beat layer
   - The timing unit for video production.
   - Each beat defines screen state, action/focus, narration, subtitle, and hold duration.

For direct live lectures, the spoken-script layer can be used as speaker notes. For YouTube video, the same layer can drive TTS rendering.

## Writing Style

Use concise Korean sentences.

Preferred style:

- Short declarative sentences.
- Use compact fragment sentences for slide titles and emphasis.
- One idea per sentence.
- Avoid long chains of clauses.
- Avoid reading slide bullets as-is.
- Avoid abstract filler.
- Use concrete verbs.
- Use transitions only when they help the audience follow the argument.

The lecture should sound like a clear spoken explanation, not a report read aloud.

Use so-called compressed sentences selectively.

Good places:

- Slide titles
- Section transitions
- One-line conclusions
- Demo captions
- Emphasis subtitles

Avoid using compressed fragments for every spoken line. Narration still needs enough connective tissue to sound natural.

Pattern:

- Slide: `강력한 AI. 낮은 사용.`
- Spoken: `AI는 강력합니다. 그런데 실제 사용은 기대보다 낮았습니다.`
- Slide: `선택은 부담. 자동화는 불안.`
- Spoken: `선택지가 많으면 부담스럽습니다. 그런데 모두 자동으로 하면 또 불안합니다.`

Example:

- Weak: `사람들은 이미 가지고 있는 데이터의 의미를 알고 싶어 하며, 그 의미를 의사결정과 보고서, 전략, 설명, 후속 질문에 활용하고 싶어 한다.`
- Better: `사람들은 데이터의 의미를 알고 싶어 한다. 그 의미를 의사결정에 쓰고 싶어 한다. 보고서와 전략에도 쓰고 싶어 한다.`

## Demo Rewrite Principle

The demos should not be app feature cards. Each demo should show an agent-design principle through a real app workflow.

Every demo should follow this structure:

1. Problem situation
   - What makes the user uncertain or blocked?

2. App execution
   - Show the actual app screen, input, run, and output.

3. Judgment point
   - Pause on the place where the app compares, validates, repairs, or chooses the level of correction.

4. Meaning
   - Explain what this shows about agent design.

5. Lecture connection
   - Tie the scene back to delegation, visible judgment, validation, or correction-level selection.

## Existing Demo Video Policy

Do not use the old demo videos as final lecture material.

Problems:

- Too many static screens.
- Narration is not natural.
- The voice often explains a concept while the screen does not show the relevant action.
- The clips feel like feature summaries, not live app demonstrations.

Use them only as rough references for what not to do.

New demos must show the app running. They should include visible input, action, result, pause point, and interpretation.

## Demo-Specific Direction

### Text Analysis Copilot

Do not present it as a text-analysis feature demo.

Show that weak results should not always lead to simple rerun. The app should help decide whether the correction belongs to wording, coding rules, codebook, theory, or the question itself.

### PR Studio Press Release

Do not present it as translation or writing automation only.

Show that a press release must be evaluated against target audience, distribution purpose, responsible wording, media context, and message structure. The important claim is that good agents need an explicit business judgment structure.

### Press Release Builder / Studio

This should be a core example of correction-level judgment.

Do not frame the problem as "translation quality" only.

The observed learning:

- First, the problem was treated as translation.
- Then the translation was edited.
- Hallucination was reduced.
- Awkward English was improved.
- Reusable phrase libraries were considered.
- But the real problem remained.
- A press release is not just translated text.
- It is a distributable message structure.

The useful reframing:

- From translation to rewriting.
- From rewriting to press-release message reconstruction.
- From sentence quality to business judgment.

Preferred lecture phrasing:

- `번역을 고친 것이 아니다. 문제의 틀을 바꾸었다.`
- `번역 문제가 아니라, 배포 가능한 메시지 구조의 문제였다.`
- `큰 철학적 패러다임 전환은 아니다. 그러나 업무 프레임은 바뀌었다.`

Avoid:

- Do not call this localization. The point is not local or cultural adaptation. The point is reframing translation as press-release rewriting and message reconstruction.

### PR Studio YouTube Feedback

Do not present it as comment summarization only.

Show that feedback should lead to the next experiment: title, thumbnail promise, description structure, target audience, or message frame. The point is not to regenerate blindly but to decide what level should change next.

### Validation Server

Do not present it as a dashboard screen.

Show the difference between plausible output and releasable output. The useful moment is when the app blocks release with concrete evidence such as base-N mismatch or cell-sum mismatch.

## PR Studio Code Improvement Backlog

These are product-level improvements PR Studio should accumulate:

- External project registry: store lecture/demo project metadata without moving large folders into this repo.
- Spoken-script editor: maintain slide text separately from natural narration.
- Narration audit: detect awkward generated phrases such as title-plus-`입니다`, raw file paths, repeated bullets, and overly long sentences.
- Beat-based renderer: render by timed beats instead of one narration block per slide.
- Demo capture adapters: support live app screen capture, cursor/focus highlights, zoom crops, and fallback recorded clips.
- Voice A/B workflow: compare Google, OpenAI, and ElevenLabs with the same spoken script.
- Screen-voice QA: sample frames and narration text together to verify that the voice mentions what is visible.
- Feedback log: record what was changed after watching a rendered lecture/demo.

## Asset Policy

Do not commit generated media or credentials.

Allowed in Git:

- Project notes
- Manifest files
- Script templates
- Renderer code
- Small sample metadata

Keep outside Git:

- MP4 files
- WAV files
- Rendered frame sequences
- Google service account JSON
- Large screenshots unless intentionally part of app assets
