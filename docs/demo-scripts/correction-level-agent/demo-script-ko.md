# Correction-Level Agent Demo

## 목적

강의 후반부에서 보여줄 데모 영상 원고입니다. 핵심 질문은 "왜 기능이 잘 돌아가는 에이전트 앱도 실제 업무에서는 쓰이지 않을까?"입니다. 답은 기능 부족만이 아니라 신뢰, 자동화, 그리고 결과의 유용성/정합성 문제라는 흐름으로 잡습니다.

## 데모 메시지

1. 사용자는 방법론 옵션을 고르는 일보다 해결 결과에 관심이 있다.
2. 그렇다고 완전 자동으로 처리하면 "정말 맞는지 확인할 수 없다"는 불안이 생긴다.
3. 다음 세대 에이전트의 핵심은 실행 자체보다 "어느 층위에서 수정해야 하는가"를 판단하고, 그 판단 근거를 보여주는 것이다.
4. 단순 가드레일은 금지/허용 규칙에 가깝다. 여기서 보여줄 것은 피드백과 결과를 보고 코드, 규칙, 이론/모형, 질문 자체 중 어느 수준을 손댈지 정하는 판단 루프다.
5. 마지막 검증은 항상 정답률만으로 끝나지 않는다. 분석 결과가 다른 변수, 설계, 해석, 사용 목적과 정합적인지 확인해야 한다.

## 영상 구성

### 1. 문제 제기

내레이션:

> 기능이 꽤 좋은 에이전트 앱을 만들어도 사람들이 실제로 쓰지 않는 경우가 있습니다. 옵션을 많이 주면 방법론을 떠안긴다고 느끼고, 자동으로 처리하면 맞는지 확인할 수 없어서 불안해합니다. 그래서 문제는 기능 수가 아니라 신뢰, 자동화, 유용성의 균형입니다.

화면:

- 강의 제목
- 신뢰, 자동화, 유용성/정합성의 3축

### 2. Text Analysis Copilot

내레이션:

> 첫 번째 예는 텍스트 분석기입니다. 여기서는 결과가 약할 때 사람이 먼저 판단하지 않고, 앱이 코딩 규칙 수정부터 시작해 필요하면 이론이나 코드북 수준까지 올라갑니다. 중요한 점은 '수정한다'가 아니라 '어느 수준을 수정할지 판단한다'는 것입니다.

시연 포인트:

- `Autonomous repair`
- `Run / redo`
- coding-rule repair -> theory/codebook review -> research-question review 흐름

### 3. PR Studio

내레이션:

> 두 번째 예는 PR Studio입니다. 유튜브 영상 품질을 한 번에 최대화하는 것이 핵심이 아닙니다. 댓글과 반응을 읽고, 제목만 바꿀지, 설명 구조를 바꿀지, 메시지 프레임이나 타깃 자체를 바꿀지 판단하는 것이 핵심입니다. 가드레일은 그 판단이 과잉 수정으로 흐르지 않게 잡아주는 장치입니다.

시연 포인트:

- `성과 수집`
- 피드백 입력
- `댓글/피드백 기반 수정 수준 판단`
- 개선 가드레일

### 4. Insight Validation Server

내레이션:

> 마지막 예는 검증 서버입니다. 여기서는 결과가 그럴듯한 문장인지가 아니라, 분석 결과가 내부적으로 맞는지, 설계와 방법이 맞는지, 사용 목적에 공개해도 되는지를 따집니다. 이것이 강의에서 말한 정합성 판단의 한 예입니다.

시연 포인트:

- `cell_sum_001` 데모 케이스
- `BASE_N_MISMATCH`
- `CELL_SUM_MISMATCH`
- `do_not_release`

### 5. 마무리

내레이션:

> 그래서 다음 세대 에이전트의 질문은 'AI가 해 주는가'에서 끝나지 않습니다. AI가 어느 수준의 문제인지 판단하고, 사람은 그 판단이 업무 목적과 사회적 맥락에 맞는지 볼 수 있어야 합니다. 완전한 정답은 없지만, 정합성 검증은 그 판단을 현실의 사용 가능성 쪽으로 끌어오는 최소한의 장치입니다.
