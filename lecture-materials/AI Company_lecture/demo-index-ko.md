## Demo 1. Text Analysis Copilot - 수정 수준 판단

- 파일: `../demos/demo-01-text-analysis-repair.mp4`
- 삽입 위치: 수정 수준 판단 설명 직후
- 핵심: 결과가 약할 때 단순 재실행이 아니라 코딩 규칙, 코드북, 이론 수준 중 어디를 고칠지 판단한다.
- 발표 포인트: "수정한다"가 아니라 "어느 수준을 수정할지 판단한다"가 차이점이다.

## Demo 2. PR Studio - 보도자료 판단 구조

- 파일: `../demos/demo-02-pr-studio-press-release.mp4`
- 보조 이미지: `../screens/assets/pr-studio-press.png`
- 삽입 위치: 보도자료/번역의 한계 설명 직후
- 핵심: 보도자료는 좋은 문장이나 직역 수정만으로 끝나지 않는다. 사실, 대상, 톤, 배포 목적, 언론 대응 포인트가 함께 맞아야 한다.
- 발표 포인트: 문장 라이브러리로 해결되지 않는 업무 판단 구조를 보여준다.

## Demo 3. PR Studio - 유튜브 피드백 기반 수정

- 파일: `../demos/demo-03-pr-studio-feedback-repair.mp4`
- 삽입 위치: 피드백 루프 설명 직후
- 핵심: 영상 품질 자체보다 댓글과 지표를 보고 제목, 썸네일, 설명 구조, 메시지 프레임, 대상 설정 중 무엇을 고칠지 판단한다.
- 발표 포인트: 가드레일은 판단을 대신하는 것이 아니라 과잉 수정과 잘못된 일반화를 막는다.

## Demo 4. Insight Validation Server - 검증 화면

- 파일: `../demos/demo-04-validation-server-ui.mp4`
- 삽입 위치: 정합성 검증 설명 직후
- 핵심: 결과가 그럴듯한지보다 공개 가능한 상태인지 별도로 검증한다.
- 발표 포인트: 생성과 검증은 분리되어야 한다.

## Demo 5. Insight Validation Server - 검증 결과

- 파일: `../demos/demo-05-validation-result.mp4`
- 삽입 위치: 검증 서버 화면 직후
- 핵심: `BASE_N_MISMATCH`, `CELL_SUM_MISMATCH`, `do_not_release`처럼 결과 공개를 막는 근거를 명시한다.
- 발표 포인트: 정합성은 모든 경우의 정답은 아니지만, 학술연구와 조사 업무에서는 최소한의 사용 가능성 검증 장치다.

## 제작 원칙

- 데모는 하나의 긴 영상으로 합치지 않는다.
- 각 데모는 30-90초 정도의 독립 클립으로 둔다.
- 한 데모에는 하나의 메시지만 담는다.
- 발표 중 "이 장면에서 무엇을 봐야 하는지"를 먼저 말한 뒤 재생한다.
- 데모 영상이 아직 없으면 슬라이드에는 이미지와 제작 대상 파일명을 넣어 둔다.
